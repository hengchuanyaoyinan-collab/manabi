# TANI_DESIGN.md — 単位取得レシピ・プラットフォーム設計

このドキュメントは、Manabi 一旦終了後の新サービス
**「単位取得レシピ・プラットフォーム」**（仮称：Tan-i）の完全設計。

クックパッド型の構造化UGCを「料理レシピ」ではなく
「通信制大学の単位取得体験記」に適用する。

---

## コアコンセプト

| クックパッド | 本サービス |
|---|---|
| 料理レシピ | 単位取得レシピ |
| 材料 | 必要なもの（教科書・時間・AI・過去問） |
| 手順 | 取得手順（レポート→試験 etc.） |
| つくれぽ | 取得報告（再現できた） |
| 完成写真 | 取得証明（任意・成績スクショ） |
| 役立った | 役に立った |
| カテゴリ | 大学・学部・科目 |

---

## 設計の3つの絶対要件

1. **構造化UGC**：自由文だけではなく、難易度・所要時間・取得方法をメタデータ化
2. **検索価値**：科目別・難易度別・期間別で絞り込める
3. **累積資産化**：レシピ × つくれぽが時間で堀になる

---

## サービス全体アーキテクチャ

```
Frontend (Next.js 15 App Router, SSR/ISR)
  ↓
Supabase
  ├ Postgres (主DB)
  ├ pgroonga (日本語全文検索)
  ├ pgvector (semantic検索)
  ├ Edge Functions
  ├ Auth
  └ Storage
  ↓
External
  ├ Anthropic Claude API (タイトル提案・要約・モデレーション)
  ├ Vercel OG (OG画像自動生成)
  └ Resend (メール)
```

**重要：MVPフェーズは Vite のままでも作れる**。
Next.js 移行は MAU 100 を超えてから判断。

## データ設計（主要テーブル）

詳細は同セッションの戦略レスポンス参照。サマリ：

- `universities` — 大学マスタ
- `faculties` — 学部
- `subjects` — 科目マスタ
- `recipes` — 単位取得レシピ（最重要）
  - 構造化メタ：`difficulty`, `duration_hours/weeks`, `method_type`, `obtained_year`, `outcome`
  - 本文：`summary`, `body_md`, `steps (jsonb)`, `tools_used (jsonb)`, `pitfalls (jsonb)`
  - 検索：`search_vector (tsvector for pgroonga)`, `embedding (pgvector)`
- `reviews` — つくれぽ（取得報告）
- `reactions` — 役に立った/やってみたい
- `comments` — コメント
- `bookmarks` — あとで読む
- `recipe_requests` — まだレシピがない科目への投稿リクエスト
- `users` — ユーザー（信頼性フラグ含む）

詳細SQLは `supabase/migrations/20260427_tani_schema.sql` を参照。

## 投稿が増える仕組み（バイラルループ）

5つのトリガー：

1. **取得直後の自動プロンプト**：取得報告から AI がレシピ下書き生成
2. **穴埋め誘導**：「あなたが取得済み × 誰もまだ書いていない」科目を可視化
3. **反応の即時通知**：「あなたのレシピが ◯◯ さんを助けました」
4. **数値の可視化**：PV / つくれぽ数 / 助けた人数（プロフィールに大きく）
5. **バッジとレベル**：投稿数 × 役立ち数で先輩レベル

4. AI 関連推薦
5. 大学別ランキング（投稿が少なくても表示）

## 実装順（推奨）

1. **シードレシピ 5本を手で書く**（コードゼロ、1〜2日）
   → `docs/SEED_RECIPES.md` のテンプレに従って
2. **既存の Vite 上に新ページとして MVP を組む**（1〜2週間）
3. **DB マイグレーション実行**（半日、`supabase/migrations/20260427_tani_schema.sql`）
4. **投稿フォーム + 閲覧ページ**（1週間）
5. **検索 + OG画像生成**（1週間）
6. **クローズドベータ：5人にDM**（1週間）

合計：6〜10週間で MVP 公開可能。

## 補足：なぜ Manabi の延長ではなくゼロから設計したか

- Manabi（マーケットプレイス）と Tan-i（ナレッジプラットフォーム）は
  **構造が根本的に違う**
- Manabi: 1対1取引、フローのお金
- Tan-i: 公共ナレッジ、累積する資産
- 同じ「教育」領域だが、データモデルもUXも別物
- Manabi のコード資産（Supabase Auth、Stripe 配管、認証）は流用可能
- ただしコアテーブル・UI は新規

## 起きた時にやること

1. このドキュメントを読む（10分）
2. `docs/SEED_RECIPES.md` のテンプレに従って自分の単位取得経験を1本書いてみる
3. 書けるか／書きたくなるか／書いて他人の役に立つかを自分で判断する
4. 書ける → 残り4本書く → 実装着手
5. 書けない → ピボット先候補を再検討

実装より、**自分で書けるかの検証が最優先**。
